# 0\. Agile Framework

# Summary

Azure DevOps has **4 primary process templates** (processes), and your impression is mostly correct: **three of them effectively have a 4-level planning hierarchy (**[\[learn.microsoft.com\]](https://learn.microsoft.com/en-us/azure/devops/boards/work-items/guidance/choose-process?view=azure-devops), [\[organicdevops.com\]](https://www.organicdevops.com/process-templates/),[\[learn.microsoft.com\]](https://learn.microsoft.com/en-us/azure/devops/boards/work-items/guidance/choose-process?view=azure-devops), [\[learn.microsoft.com\]](https://learn.microsoft.com/en-us/azure/devops/boards/work-items/guidance/cmmi-process-workflow?view=azure-devops), [\[organicdevops.com\]](https://www.organicdevops.com/process-templates/))

## DevOps Process Template Comparison Matrix

| Dimension | Agile | Scrum | CMMI |
| --- | --- | --- | --- |
| Intended methodology | Generic Agile | Scrum Framework | Process-heavy / governance-oriented |
| Portfolio Level 1 | Epic | Epic | Epic |
| Portfolio Level 2 | Feature | Feature | Feature |
| Backlog Level | User Story | Product Backlog Item (PBI) | Requirement |
| Execution Level | Task | Task | Task |
| Typical Hierarchy | Epic → Feature → User Story → Task | Epic → Feature → PBI → Task | Epic → Feature → Requirement → Task |
| Bug Tracking | Bug | Bug | Bug |
| Risk Tracking | No | No | Yes |
| Change Request Tracking | No | No | Yes |
| Impediment Tracking | Issue | Impediment | Issue |
| Governance Maturity | Medium | Medium | High |
| Reporting Complexity | Moderate | Moderate | Highest |
| Best For | Most software teams | Pure Scrum teams | Regulated enterprises |

## Translation Matrix

These are essentially equivalent concepts:

| Business Concept | Agile | Scrum | CMMI |
| --- | --- | --- | --- |
| User Need | User Story | Product Backlog Item | Requirement |
| Delivery Capability | Feature | Feature | Feature |
| Strategic Initiative | Epic | Epic | Epic |
| Work Package | Task | Task | Task |

The three processes are largely vocabulary changes at the backlog layer:

| Business Meaning | Agile | Scrum | CMMI |

| Deliverable Requirement | User Story | PBI | Requirement |
| Work Package | Task | Task | Task |

# Assessment for Data Teams

As a Lead Data Engineer or Data Platform Architect, I typically recommend:

| Process | Recommendation | Confidence |
| --- | --- | --- |
| Agile | Best overall choice | 95% |
| Scrum | Good if organization mandates Scrum ceremonies | 90% |
| CMMI | Only for highly regulated environments (banking, government, healthcare) | 90% |

# Assessment for a data environment

Considering work with:

- Databricks
- Snowflake
- dbt
- Terraform
- DataOps
- GitHub/Azure DevOps
- Banking data platforms

I would recommend:

Epic

 └── Feature

      └── User Story

            └── Task

using the **Agile process**.

Reason:

- Better alignment with DataOps and platform engineering.
- User Stories map naturally to data products.
- Features map well to domains (Customer 360, Payments, Risk, Lending).
- Epics map well to platform modernization initiatives.
- Simpler than CMMI while still supporting enterprise reporting.

For banking environments that require formal controls, change requests, and audit trails, CMMI becomes more attractive, but I'd still start with Agile unless your PMO explicitly requires CMMI.[\[learn.microsoft.com\]](https://learn.microsoft.com/en-us/azure/devops/boards/work-items/guidance/choose-process?view=azure-devops), [\[learn.microsoft.com\]](https://learn.microsoft.com/en-us/azure/devops/boards/work-items/guidance/cmmi-process-workflow?view=azure-devops)

A useful mental model is:

| DevOps | Data Platform Example |
| --- | --- |
| Epic | Customer Analytics Modernization |
| Feature | Customer Gold Layer |
| User Story | Create Canonical Customer Entity |
| Task | Build dbt model, tests, documentation |

# Recommendation for Banking Lakehouse

If you're organizing work around:

```
Business Domain
 → Data Product
   → Data Entity
     → Engineering Tasks
```

I would map Azure DevOps Agile as:

```
Epic = Banking Domain
Feature = Data Product
User Story = Canonical Entity / Business Capability
Task = Engineering Work
```

Example:

```
Epic: Customer Domain
Feature: Customer 360 Data Product
User Story: Deliver Canonical Customer Entity
Tasks:
Build dbt model
Add data quality tests
Create semantic layer mapping
Publish documentation
```

# Delivery Increment with ScrumBan

So the real comparison is:

| Dimension | Agile Process | Scrum Process | CMMI Process | Kanban Method | Scrumban |
| --- | --- | --- | --- | --- | --- |
| Azure DevOps Native Template | ✅ | ✅ | ✅ | ❌ | ❌ |
| Epic Level | ✅ | ✅ | ✅ | Optional | Optional |
| Feature Level | ✅ | ✅ | ✅ | Optional | Optional |
| Backlog Level | User Story | PBI | Requirement | Work Item | Work Item |
| Task Level | ✅ | ✅ | ✅ | Optional | ✅ |
| Sprint-Based | Optional | Mandatory | Optional | No | Usually Optional |
| Continuous Flow | Possible | Limited | Possible | Yes | Yes |
| WIP Limits | Optional | Rare | Rare | Core Practice | Core Practice |
| Velocity Tracking | Good | Excellent | Good | Weak | Moderate |
| Governance | Medium | Medium | High | Low | Medium |
| PMO Reporting | Excellent | Excellent | Excellent | Moderate | Good |
| Enterprise Architecture Alignment | Excellent | Good | Excellent | Moderate | Good |
| Data Platform Alignment | Excellent | Good | Good | Very Good | Excellent |
